# ADR-003: Выбор embedding-модели и размерности для GraphRAG-системы

**Статус:** Proposed  
**Дата:** 2026-04-24  
**Автор:** AI-Architect  

## Контекст

GraphRAG-система строится на корпусе научных статей arXiv по теме *LLM for Time Series Forecasting*. В ADR-002 в качестве векторной базы данных выбран Qdrant, а основной dense retrieval строится по синтетическому полю `retrieval_text`, которое объединяет название статьи, путь раздела, `summary` и `keywords`.

Корпус небольшой: сейчас 163 чанка из 2 статей, ближайшее расширение ожидается до 2-3 тысяч чанков. Поэтому стоимость первичной индексации и объём хранения dense-векторов не являются главным ограничением. Более важны:

- качество семантического поиска по научному английскому тексту;
- устойчивость к формулировкам запросов про методы, задачи, датасеты, метрики и техники;
- воспроизводимость индексации;
- совместимость с Qdrant и последующим hybrid search;
- возможность переиндексации при смене модели.

Запросы к системе будут смешанными: часть запросов семантическая ("какие подходы используют LLM как frozen backbone"), часть терминологически точная ("GPT-2", "ETTm1", "MSE", "rsLoRA"). Точные совпадения закрываются sparse/hybrid search в ADR-004, но dense-модель всё равно должна хорошо покрывать научные парафразы.

## Решение

**Использовать OpenAI `text-embedding-3-large` с размерностью `3072`** для основного dense-вектора `dense_retrieval`.

Параметры dense-представления:

| Параметр | Значение |
| --- | --- |
| Модель | `text-embedding-3-large` |
| Размерность | `3072` |
| Qdrant vector name | `dense_retrieval` |
| Distance | `Cosine` |
| Embedding input | `retrieval_text` |
| Формат | `float` |

При вызове OpenAI Embeddings API параметр `dimensions` не передаётся: используется полная размерность модели. Это фиксирует максимальное качество dense-представления на первом этапе. Уменьшение размерности, например до `1024`, допускается только отдельным решением после retrieval-evaluation и полной переиндексации коллекции.

## Детали реализации

### Поле для embedding

Dense embedding строится по `retrieval_text`, а не по полному `text` и не по одному `summary`.

Функция построения:

```python
section_path = " > ".join(chunk["section_hierarchy"])
retrieval_text = "\n".join([
    f"Title: {chunk['article_meta']['title']}",
    f"Section: {section_path}",
    f"Summary: {chunk['summary']}",
    f"Keywords: {', '.join(chunk['keywords'])}",
])
```

`retrieval_text` сохраняется в payload для воспроизводимости. При изменении этой функции требуется переиндексация dense-векторов.

### Метаданные индексации

В payload или отдельном manifest-файле загрузки фиксируются:

```json
{
  "embedding_vector_name": "dense_retrieval",
  "embedding_provider": "openai",
  "embedding_model": "text-embedding-3-large",
  "embedding_dim": 3072,
  "embedding_input_field": "retrieval_text",
  "embedding_distance": "Cosine"
}
```

Эти поля нужны, чтобы не смешать в одной коллекции векторы разных моделей, размерностей или функций подготовки текста.

### Обработка длинных чанков

Текущий максимальный размер чанка — 4659 токенов, но dense embedding строится по компактному `retrieval_text`, поэтому ограничение входа embedding-модели не должно быть проблемой. Полный `text` остаётся в payload и используется на этапах sparse search, reranking, генерации ответа и извлечения сущностей.

### Запросы

Пользовательский запрос кодируется той же моделью `text-embedding-3-large`. На первом этапе embedding строится по исходному запросу без LLM-переписывания. Если позже появится query rewriting, оно должно быть включено в evaluation как отдельный retrieval-профиль.

## Причины выбора

1. **Качество важнее экономии на текущем размере корпуса.** При 2-3 тысячах чанков разница в стоимости индексации между `text-embedding-3-small` и `text-embedding-3-large` мала относительно риска потерять релевантные научные фрагменты.
2. **Полная размерность упрощает baseline.** `3072` даёт понятную стартовую точку без дополнительного компромисса по Matryoshka/shortened embeddings.
3. **Совместимость с Qdrant.** Qdrant хранит dense-вектор как обычный named vector, а размерность фиксируется в конфигурации коллекции.
4. **Хорошая роль в hybrid retrieval.** Dense-вектор отвечает за семантическую близость и парафразы, sparse-вектор ADR-004 — за точные термины, модели, датасеты и численные обозначения.
5. **Переиндексация остаётся дешёвой.** Пока корпус мал, можно безболезненно пересоздать коллекцию при смене модели или размерности.

## Рассмотренные альтернативы

| Альтернатива | Причина отклонения |
| --- | --- |
| OpenAI `text-embedding-3-small`, `1536` | Дешевле и быстрее, но качество ниже. Экономия не критична для текущего корпуса, а dense recall важен для научных запросов. |
| OpenAI `text-embedding-3-large`, `1024` через `dimensions` | Потенциально хороший компромисс для большого корпуса, но на текущем этапе нет собственной retrieval-evaluation, подтверждающей отсутствие потерь. Оставлено как оптимизация после baseline. |
| Локальные encoder-модели семейства E5/BGE/GTE | Убирают зависимость от внешнего API, но добавляют эксплуатационные расходы: выбор модели, окружение, GPU/CPU latency, версионирование и качество на научном корпусе. Для публичных arXiv-статей внешний API допустим. |
| Embedding по полному `text` | Полный текст содержит таблицы, формулы, LaTeX и артефакты конвертации, которые могут зашумить dense-вектор. Полный текст лучше использовать для sparse search и reranking. |
| Отдельные dense-векторы для `summary`, `text`, `keywords` | Может повысить качество, но усложняет схему и evaluation. На первом этапе достаточно одного dense-вектора по `retrieval_text`. |

## Последствия

### Положительные

- Dense retrieval получает более сильную модель для семантических запросов.
- Размерность и имя вектора явно зафиксированы до создания Qdrant-коллекции.
- Сохраняется простая схема: один dense-вектор на чанк.
- Решение совместимо с гибридным поиском из ADR-004.

### Риски и требования

- Появляется зависимость от OpenAI API и доступности ключа.
- При смене модели, размерности или функции `retrieval_text` требуется полная переиндексация.
- Для контроля качества нужна retrieval-evaluation: dense-only, sparse-only и hybrid должны сравниваться на одном наборе тестовых запросов.
- Если корпус вырастет на порядки, нужно повторно оценить размерность `3072` против `1024` или локальных моделей.
