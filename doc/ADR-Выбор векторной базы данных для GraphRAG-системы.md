# ADR-002: Выбор векторной базы данных для GraphRAG-системы

**Статус:** Proposed  
**Дата:** 2026-04-24  
**Автор:** AI-Architect  

## Контекст

Разрабатывается GraphRAG-система, базой знаний для которой служат научные статьи arXiv по теме *LLM for Time Series Forecasting*. Документы прошли стадию чанкинга и обогащения: сформированы структурированные чанки с метаданными, описанные в `chunks/doc/Структура и содержимое чанков.md`.

Текущий корпус `chunks/dic_chunks.json` содержит:

- **163 чанка** из **2 статей**;
- средний размер чанка: около **517 токенов**;
- максимальный размер чанка: **4659 токенов**;
- типы чанков: `prose`, `formula_heavy`, `table_with_prose`, `table`;
- разделы: `Abstract`, `Introduction`, `RelatedWork`, `Methodology`, `Experiments`, `Appendix_Results`, `Appendix_Metrics` и другие.

Ожидаемый объём ближайшего расширения корпуса — **2-3 тысячи чанков**. Даже при таком объёме полный текст чанков остаётся небольшим по размеру и может храниться непосредственно в payload векторной БД без существенной нагрузки.

Каждый чанк содержит:

- `text` — полный исходный фрагмент статьи с формулами, таблицами и markdown-разметкой;
- `summary` — краткое естественно-языковое описание чанка;
- `keywords` — предварительно извлечённые сущности, методы, датасеты, метрики и домены;
- `section`, `section_hierarchy`, `chunk_type`;
- `article_meta` — вложенный объект с названием, авторами, годом и arXiv ID;
- `index`, `token_count`, `chunk_id`, `article_id`.

Стратегия поиска, заложенная в дизайн, — многоступенчатый retrieval:

1. **Candidate retrieval**: dense vector search по embedding поля `retrieval_text`, затем hybrid dense+sparse search по ADR-004.
2. **Keyword / metadata filtering** по `keywords`, `section`, `section_hierarchy`, `chunk_type`, `article_meta.year`, `article_id` и другим payload-полям.
3. **Reranking** с подачей полного `text` лучших кандидатов в cross-encoder или LLM.

Необходимо выбрать векторную базу данных, которая поддерживает хранение плотных и sparse-векторов, полноценную фильтрацию по JSON payload, простую интеграцию с Python-пайплайном и hybrid search.

## Решение

**Использовать Qdrant** в качестве векторной базы данных. Развёртывание на этапе прототипа — локально через Docker / Docker Compose. При необходимости допускается Qdrant Cloud.

Qdrant используется как самодостаточное хранилище retrieval-слоя: в одной точке хранятся dense/sparse vector-представления и полный payload чанка, включая `text`, `summary` и служебные поля. Отдельное KV-хранилище для `text` на первом этапе не используется, чтобы избежать рассинхронизации данных и упростить отладку.

## Детали реализации

### Коллекция

Создаётся коллекция `chunks`.

Основной dense-вектор хранится как named vector `dense_retrieval`.

Размерность dense-вектора определяется ADR-003:

- модель: OpenAI `text-embedding-3-large`;
- размерность: `3072`;
- поле-источник: `retrieval_text`;
- метрика сходства: `Cosine`.

Sparse-вектор `sparse_bm25` и гибридная стратегия поиска определяются ADR-004. Если на первом техническом шаге загружается только dense-вектор, коллекция всё равно должна проектироваться с учётом последующего добавления sparse-представления или пересоздаваться перед продуктивной индексацией.

Embedding-модель фиксируется до создания коллекции. При смене модели коллекция пересоздаётся и наполняется заново.

### Поле для embedding

Dense embedding строится не по чистому `summary`, а по синтетическому полю `retrieval_text`.

`retrieval_text` конструируется при загрузке чанков:

```python
section_path = " > ".join(section_hierarchy)
retrieval_text = "\n".join([
    f"Title: {title}",
    f"Section: {section_path}",
    f"Summary: {summary}",
    f"Keywords: {', '.join(keywords)}",
])
```

где:

- `title = article_meta.title`;
- `section_path` — полный путь раздела из `section_hierarchy`;
- `summary` — краткое описание чанка;
- `keywords` — список извлечённых сущностей.

`retrieval_text` хранится в payload. Это нужно для воспроизводимости embedding, отладки retrieval и последующей переиндексации.

Причина выбора `retrieval_text`: чистый `summary` может опустить точные названия моделей, метрик, датасетов и техник. Добавление заголовка статьи, точного раздела и `keywords` повышает recall для научных запросов вида "какие методы используют GPT-2", "где сравнивается MSE на ETT", "какие подходы используют decomposition".

### Payload

Payload каждой точки сохраняет исходную вложенную структуру чанка и дополнительные служебные поля:

```json
{
  "chunk_id": "arxiv_2403_05798_chunk_007",
  "article_id": "arxiv_2403_05798",
  "article_meta": {
    "title": "S2IP-LLM: Semantic Space Informed Prompt Learning with LLM for Time Series Forecasting",
    "authors": ["Pan", "Jiang", "Garg", "Schneider", "Nevmyvaka", "Song"],
    "year": 2024,
    "arxiv_id": "2403.05798"
  },
  "section": "Methodology",
  "section_hierarchy": ["Methodology", "3.2. Time Series Tokenization"],
  "chunk_type": "prose",
  "index": 7,
  "text": "...",
  "summary": "...",
  "retrieval_text": "...",
  "keywords": ["S2IP-LLM", "GPT-2", "tokenization", "decomposition"],
  "token_count": 493,
  "chunker_version": "0.1.0",
  "config_hash": "2f09dc9dfd28",
  "embedding_vector_name": "dense_retrieval",
  "embedding_model": "text-embedding-3-large",
  "embedding_dim": 3072
}
```

Поле `text` хранится в Qdrant payload на первом этапе. Это делает каждую точку самодостаточной: результат поиска сразу содержит исходный фрагмент для reranking, генерации ответа, трассировки и ручной проверки.

Служебные поля `chunker_version`, `config_hash`, `embedding_model`, `embedding_dim` не обязательно индексировать, но они должны храниться для воспроизводимости.

### Индексы payload

Создаются индексы на поля, используемые в фильтрах:

| Поле | Тип индекса | Назначение |
| --- | --- | --- |
| `chunk_id` | keyword | Точное получение чанка, отладка |
| `article_id` | keyword | Фильтрация по статье и восстановление соседей |
| `article_meta.year` | integer | Фильтрация по году |
| `article_meta.arxiv_id` | keyword | Поиск по arXiv ID |
| `section` | keyword | Фильтрация по крупному разделу |
| `section_hierarchy` | keyword | Фильтрация по подразделам |
| `chunk_type` | keyword | Разделение прозы, таблиц и формул |
| `keywords` | keyword | Точные совпадения сущностей |
| `index` | integer | Восстановление соседних чанков |

Qdrant поддерживает фильтрацию по вложенным JSON-полям через точечную нотацию. Поэтому `article_meta` сохраняется как вложенный объект, без денормализации в плоские поля.

Пример фильтра:

```json
{
  "must": [
    {
      "key": "article_meta.year",
      "range": { "gte": 2023 }
    },
    {
      "key": "section",
      "match": { "value": "Methodology" }
    },
    {
      "key": "keywords",
      "match": { "value": "GPT-2" }
    }
  ]
}
```

### Сценарии поиска

#### Vector search с нативным фильтром

Если пользовательский запрос содержит ограничения по году, разделу, типу чанка, статье или сущностям, эти ограничения передаются в Qdrant как `query_filter`.

Qdrant должен возвращать только точки, удовлетворяющие фильтру. Фильтрация после получения top-N не является базовым сценарием, потому что она может удалить релевантные документы из уже усечённой выдачи.

Пример:

1. Пользователь спрашивает: "Какие методы на базе GPT-2 применяются в Methodology-разделах после 2023 года?"
2. Строится dense embedding пользовательского запроса.
3. Выполняется vector search по `retrieval_text` с фильтром:
   - `keywords = "GPT-2"`;
   - `section = "Methodology"`;
   - `article_meta.year >= 2023`.

#### Keyword-first

Если запрос явно указывает на конкретные сущности, сначала задаётся фильтр по `keywords`, `section`, `chunk_type`, `article_meta.year` и другим полям, затем внутри отфильтрованного подмножества выполняется vector search.

Этот сценарий полезен для запросов вида:

- "STL-декомпозиция";
- "результаты по MSE";
- "датасет ETT";
- "rsLoRA в LLM4TS".

#### Oversampling

Постфильтрация допустима только как исключение, если фильтр нельзя выразить средствами Qdrant или он применяется внешней логикой. В этом случае требуется oversampling: например, запросить top-200 кандидатов, чтобы после внешней фильтрации осталось top-20.

Базовая архитектура предпочитает нативные payload-фильтры Qdrant.

#### Reranking

Qdrant возвращает `chunk_id`, score и payload, включая `text`. Далее top-K кандидатов передаются в cross-encoder или LLM-as-a-reranker. На этом этапе используется полный `text`, а не `retrieval_text`, потому что reranker должен видеть таблицы, формулы, численные результаты и контекст.

При необходимости контекст расширяется соседними чанками через `article_id + index`.

## Причины выбора Qdrant

1. **Развитая фильтрация:** Qdrant поддерживает булевы выражения над payload, вложенные JSON-поля и массивы строк. Это хорошо соответствует полям `keywords`, `section_hierarchy`, `article_meta.year`, `chunk_type`.
2. **Простота прототипа:** один Docker-контейнер, Python-клиент, REST/gRPC API, быстрая загрузка небольшого корпуса.
3. **Самодостаточный retrieval-слой:** Qdrant может хранить полный payload чанка, включая `text`, что упрощает отладку и исключает дополнительное хранилище на первом этапе.
4. **Гибридное расширение:** Qdrant поддерживает named vectors и sparse vectors, поэтому выбранная база совместима со стратегией ADR-004: dense+sparse hybrid search с BM25 и последующим reranking.
5. **Производительность с запасом:** текущие 163 чанка и ожидаемые 2-3 тысячи чанков не создают нагрузки. При росте корпуса Qdrant поддерживает шардирование, репликацию и payload indexes.
6. **Хорошая граница ответственности:** Qdrant отвечает за candidate retrieval и metadata filtering. Финальная оценка качества ответа остаётся за reranker и LLM.

## Рассмотренные альтернативы

| Альтернатива | Причина отклонения |
| --- | --- |
| **Chroma** | Хорошее решение для быстрых прототипов, но менее выразительно для сложной фильтрации и гибридного поиска. Для текущей GraphRAG-схемы с активным использованием payload-фильтров Qdrant подходит лучше. |
| **Weaviate** | Мощная база с GraphQL и встроенной векторизацией, но более ресурсоёмкая и сложная в конфигурации. Встроенная векторизация не нужна, так как embedding генерируется отдельным пайплайном. |
| **pgvector (PostgreSQL)** | Хороший выбор, если Postgres уже является основной БД проекта. В текущем проекте это добавляет SQL-схему, ручную настройку полнотекстового поиска, индексов и ранжирования. Для специализированного vector retrieval Qdrant проще. |
| **FAISS / ScaNN** | Подходят как локальные индексы в памяти, но не дают полноценного persistent API, payload-фильтрации и удобной синхронизации с метаданными. Фактически пришлось бы строить собственную мини-БД вокруг индекса. |

## Последствия

### Положительные

- Векторная коллекция самодостаточна: payload содержит всё, что нужно для поиска, reranking, генерации ответа и отладки.
- Нет риска рассинхронизации между Qdrant и отдельным KV-хранилищем текста.
- Dense embedding по `retrieval_text` повышает recall по именованным сущностям, метрикам, датасетам и разделам.
- Нативные payload-фильтры Qdrant уменьшают риск потери релевантных документов при постфильтрации.
- Схема payload остаётся совместимой с текущим `dic_chunks.json`, включая вложенный `article_meta`.
- Архитектура совместима с sparse/hybrid search в той же Qdrant-коллекции.

### Риски и требования

- Нужно явно зафиксировать функцию построения `retrieval_text`, чтобы переиндексация была воспроизводимой.
- Нужно хранить `embedding_model` и `embedding_dim` в payload или metadata загрузки, чтобы исключить смешивание векторов разных моделей.
- При смене embedding-модели требуется пересоздать коллекцию и переиндексировать чанки.
- Индексы payload должны соответствовать реальным путям полей: например, `article_meta.year`, а не плоский `year`.
- `keywords` являются предварительным приближением entity extraction, а не финальным графом знаний. Для точного GraphRAG всё равно нужен отдельный этап извлечения сущностей и отношений.
- Для запросов по точным численным значениям внутри таблиц одного dense embedding по `retrieval_text` недостаточно. Базовый sparse/full-text слой определён в ADR-004; при необходимости дополнительно потребуется специализированный индекс по извлечённым результатам экспериментов.

## Связанные решения

- ADR-001 — стратегия чанкинга и структура обогащённого чанка.
- ADR-003 — выбор embedding-модели и её размерности.
- ADR-004 — стратегия sparse/hybrid search.
