# План поэтапной реализации ADR-005

**Статус:** Draft  
**Дата:** 2026-05-22  
**Область:** реализация слоя графа знаний по ADR-005 поверх уже работающего retrieval-слоя Qdrant  

## Цель

Построить воспроизводимый слой графа знаний для GraphRAG-системы по корпусу из `chunks 2/docs/dic_chunks.json`, не ломая уже выбранную архитектуру retrieval:

1. Развернуть локальный Neo4j-контур рядом с Qdrant.
2. Зафиксировать контракт графовых сущностей и связей по `graph-ontology/ontology.md`.
3. Реализовать идемпотентную загрузку `Article`, `Chunk`, базовых сущностей, `CONTAINS`, `MENTIONS`.
4. Связать расширение графовым контекстом с существующим сценарием `Qdrant-first`.
5. Подготовить базу для последующего извлечения доменных связей второго уровня.
6. Зафиксировать операционный контур запуска, проверки и переиндексации графа.

## Покрываемые решения

| Решение | Что уже зафиксировано | Что должно быть реализовано |
| --- | --- | --- |
| ADR-002 | Qdrant остаётся векторной БД | Графовый слой не дублирует dense/sparse поиск и связывается с Qdrant через `chunk_id` / `article_id` |
| ADR-003 | Dense embedding `text-embedding-3-large`, `3072` | Neo4j не хранит embeddings и не подменяет retrieval-слой |
| ADR-004 | Dense + sparse + hybrid retrieval | Расширение графовым контекстом подключается после retrieval, а не вместо него |
| ADR-005 | Neo4j выбран как графовая БД | Локальная инфраструктура, схема, загрузка графа, диагностические команды |
| `graph-ontology/ontology.md` | Онтология из 17 классов | Пошаговое отображение классов и связей в графе свойств |
| `chunks/doc/Структура и содержимое чанков.md` | Схема чанков v1.2.0 | Использование `question_types`, `keywords`, `summary`, `article_meta.*` в графовом контуре |

## Предпосылки

- Корпус `chunks 2/docs/dic_chunks.json` валиден и является текущим источником данных.
- Qdrant-слой уже адаптирован под формат чанков v1.2.0.
- Команды `validate-chunks`, `create-collection`, `index-chunks --dense --sparse` рассматриваются как рабочий базовый контур retrieval.
- `graph-ontology/ontology.md` является источником истины для классов и типов доменных связей.
- На первом этапе Neo4j не должен становиться второй копией Qdrant и не должен хранить полный `text` всех чанков без явной необходимости.

## Этап 0. Архитектурные границы и конфигурация

**Цель:** формально закрепить, где заканчивается retrieval и где начинается слой графа знаний.

Работы:

1. Добавить в конфигурацию отдельный раздел `graph`:
   - `uri`;
   - `user`;
   - `password_env`;
   - `database`;
   - `batch_size`;
   - `enabled`.
2. Зафиксировать переменные окружения для Neo4j в `.env.example`.
3. Определить набор будущих CLI-команд:
   - `check-graph`;
   - `create-graph-schema`;
   - `index-graph`;
   - `expand-chunks`.
4. Явно закрепить правило: retrieval идёт через Qdrant, а расширение графовым контекстом выполняется после него и опирается на `chunk_id`.

Артефакты:

- `config/retrieval.yaml` или отдельный графовый конфиг;
- `.env.example`;
- базовый каркас CLI для графового контура;
- обновлённая документация запуска.

Критерии готовности:

- Конфигурация Neo4j читается из проекта так же предсказуемо, как конфигурация Qdrant.
- В кодовой базе нет двусмысленности, какой слой отвечает за поиск, а какой за обход графа.

2. Зафиксировать:
   - версию образа;
   - порты `7474` и `7687`;
   - volume для данных;
   - credentials через переменные окружения.
3. Добавить проверку доступности Neo4j из приложения.
4. Подготовить короткий сценарий запуска рядом с уже существующим Qdrant.

Артефакты:

- `docker-compose.yml`;
- модуль подключения к Neo4j;
- команда `check-graph`.

Критерии готовности:

- Neo4j поднимается локально одной командой.
- Приложение успешно устанавливает Bolt-соединение.
- Ошибки подключения диагностируются понятным сообщением.

## Этап 2. Контракт графовых данных и правила нормализации

**Цель:** зафиксировать, какие сущности и свойства появляются в графе на первом техническом запуске.

Работы:

1. Описать минимальные графовые сущности первого этапа:
   - `Article`;
   - `Chunk`;
   - `Method`;
   - `Architecture`;
   - `Technique`;
   - `Dataset`;
   - `Benchmark`;
   - `Metric`;
   - `Domain`;
   - `Task`;
   - `TaxonomyCategory`.
2. Зафиксировать канонические идентификаторы:
   - `Article.article_id = article_id`;
   - `Chunk.chunk_id = chunk_id`;
   - `Entity.id = "<Label>:<normalized_name>"`.
3. Зафиксировать минимальный набор свойств:
   - для `Article`: `article_id`, `title`, `year`, `arxiv_id`;
   - для `Chunk`: `chunk_id`, `article_id`, `section`, `index`, `chunk_type`, `question_types`, `token_count`, `summary`;
   - для доменных сущностей: `id`, `name`, `source`.
4. Зафиксировать provenance на связях:
   - `source_chunk_id`;
   - `article_id`;
   - `extraction_source`;
   - `question_types`.
5. Описать правила классификации `keywords -> ontology class`.
6. Явно отделить `question_types` Eval API от `TaxonomyCategory` онтологии.

Артефакты:

- модуль графовых моделей или схемы импорта;
- таблица нормализации `keywords -> label`;
- документ или кодовый словарь спорных и неоднозначных сущностей.

Критерии готовности:

- Для каждого `keyword` существует понятный маршрут: создать узел нужного класса, отложить, либо пропустить с диагностикой.
- Повторная загрузка тех же чанков не создаёт новые сущности с другим идентификатором.

   - `Dataset.id`;

2. Вынести Cypher-операции создания схемы в отдельный модуль.
3. Добавить идемпотентную команду `create-graph-schema`.
4. Подготовить минимальные служебные операции:
   - очистка графа только по явному подтверждению;
   - чтение статистики по labels и relationships;
   - выборка тестовых узлов по `chunk_id` и `article_id`.

Артефакты:

- `src/vector_db_rag/graph_schema.py` или эквивалент;

- тесты или проверяемые шаблоны Cypher для схемы.

Критерии готовности:

- Схема создаётся повторно без ошибок.
- База отклоняет дубликаты по ключевым идентификаторам.

## Этап 4. Минимальный импорт графа из корпуса чанков

**Цель:** получить первый рабочий граф по корпусу v1.2.0 без доменного переусложнения.

Работы:

1. Прочитать `chunks 2/docs/dic_chunks.json`.
2. Создать или обновить узлы `Article`.
3. Создать или обновить узлы `Chunk`.
4. Создать `(:Article)-[:CONTAINS]->(:Chunk)`.
5. Нормализовать `keywords` и создать узлы сущностей базовых классов.
6. Создать `(:Chunk)-[:MENTIONS]->(:Entity)` с provenance.
7. Выполнять загрузку пакетами через `UNWIND` + `MERGE`.
8. Сформировать отчёт импорта:
   - число статей;
   - число чанков;
   - число сущностей по классам;
   - число `MENTIONS`;
   - число пропущенных или неоднозначных `keywords`.

Артефакты:

- `src/vector_db_rag/graph_store.py` или аналогичный модуль;
- `src/vector_db_rag/graph_indexing.py`;
- команда `index-graph`;
- отчёт импорта.

Критерии готовности:

- В Neo4j присутствуют все 12 статей и все 138 чанков.
- Все связи `Article -> Chunk` созданы.
- Граф импортируется повторно без роста числа дублей.

## Этап 5. Диагностика и структурная валидация графа

**Цель:** быстро обнаруживать нарушения контракта между чанками, онтологией и графом.

Работы:

1. Реализовать команду `check-graph`.
2. Добавить проверки:
   - нет `Chunk` без `Article`;
   - нет `Article` без чанков;
   - нет сущностей без входящих `MENTIONS`, если они созданы из корпуса;
   - `article_id` и `chunk_id` совпадают с идентификаторами в Qdrant payload;
   - `question_types` не теряются при импорте.
3. Добавить выборочные контрольные Cypher-запросы для ручной проверки:
   - по одной статье;
   - по одному типу сущностей;
   - по одному `chunk_id`.
4. Подготовить человекочитаемый отчёт диагностики.

Артефакты:

- диагностические запросы;
- команда `check-graph`;
- тесты на преобразование входных данных в графовые строки импорта.

Критерии готовности:

- Можно одной командой проверить согласованность корпуса и графа.
- Основные нарушения показываются в явном виде, а не скрываются внутри Cypher-ошибок.

## Этап 6. Интеграция Neo4j с текущим retrieval-контуром

**Цель:** реализовать базовый сценарий `Qdrant-first`, описанный в ADR-005.

Работы:

1. Оставить Qdrant точкой входа для поиска релевантных чанков.
2. После retrieval извлекать `chunk_id` найденных кандидатов.
3. Выполнять расширение графовым контекстом в Neo4j по этим `chunk_id`.
4. Возвращать из графа:
   - упомянутые сущности;
   - ближайшие доменные связи;
   - дополнительный структурный контекст по статье.
5. Зафиксировать формат ответа для следующего слоя:
   - `retrieved_chunks`;
   - `graph_entities`;
   - `graph_relations`;
   - `graph_paths`.
6. Добавить отдельный режим запроса, например `query --expand-graph`.

Артефакты:

- модуль расширения графовым контекстом;
- расширение CLI `query`;
- тесты сборки ответа из Qdrant + Neo4j.

Критерии готовности:

- Один пользовательский запрос может вернуть не только чанки, но и связанный графовый контекст.
- Расширение по графу не ломает действующие retrieval-профили.

## Этап 8. Проверка на сценариях Eval API

**Цель:** доказать, что графовый слой помогает retrieval и будущему GraphRAG-ответу, а не существует отдельно от задачи.

Работы:

1. Составить набор проверок по таксономии `А–М` и `Спец`.
2. Для каждого сценария фиксировать:
   - исходный запрос;
   - целевую статью;
   - ожидаемые `chunk_id`;
   - ожидаемые сущности и связи.
3. Проверить два уровня:
   - retrieval находит нужный чанк;
   - расширение графовым контекстом добавляет полезный структурный контекст.
4. Отдельно проверить вопросы по:
   - методам;
   - архитектурам;
   - датасетам;
   - метрикам;
   - численным результатам;
   - обзорным категориям.

Артефакты:

- набор сценариев проверки;
- отчёт по качеству расширения графовым контекстом;
- список пробелов в `keywords` и нормализации.

Критерии готовности:

- Графовый слой демонстрирует практическую пользу хотя бы на пилотном наборе вопросов.
- Есть список точек, где требуется усиление сущностного извлечения или доменных связей.

## Приоритеты реализации

1. Сначала закрепить инфраструктуру и схему Neo4j.
2. Затем получить минимальный идемпотентный импорт из корпуса чанков.
3. После этого добавить диагностику и только потом встраивать расширение графовым контекстом в пользовательский путь запроса.
4. Доменные связи второго уровня вводить после того, как базовый граф стабилен и проверяем.
5. Проверку полезности на вопросах Eval API проводить на уже работающем сценарии `Qdrant-first + расширение графовым контекстом`.

## Основные риски

| Риск | Где проявится | Смягчение |
| --- | --- | --- |
| Рассинхронизация Qdrant и Neo4j по `chunk_id` / `article_id` | Retrieval и расширение графовым контекстом | Единый источник данных `chunks 2/docs/dic_chunks.json`, явные проверки в `check-graph` |
| Смешение `question_types` и `TaxonomyCategory` | Модель графа и ответы | Хранить `question_types` как свойство `Chunk`, а не как узлы таксономии |
| Шумная или неполная классификация `keywords` | `MENTIONS` и доменные связи | Явная таблица нормализации, журнал пропусков, пилотная ручная проверка |
| Дублирование данных между Qdrant и Neo4j | Хранение и сопровождение | Не переносить полный `text` чанка в Neo4j на первом этапе |
| Неидемпотентный импорт | Повторные загрузки и отладка | `MERGE`, уникальные ограничения, отчёт по счётчикам до и после |
| Преждевременное усложнение доменных связей | Сроки и надёжность | Сначала минимальный граф `Article/Chunk/MENTIONS`, потом обогащение |

## Definition of Done

ADR-005 можно считать реализованным на минимально полезном уровне, когда выполняются все условия:

- Neo4j поднимается из репозитория рядом с Qdrant.
- Конфигурация графового слоя читается из проекта.
- Уникальные ограничения графовой схемы создаются автоматической командой.
- Корпус `chunks 2/docs/dic_chunks.json` импортируется в виде `Article`, `Chunk`, базовых сущностей, `CONTAINS`, `MENTIONS`.
- Все `chunk_id` и `article_id` в Neo4j совпадают с retrieval-слоем.
- Графовая диагностика умеет выявлять сироты, дубли и потери provenance.
- Пользовательский запрос может получить расширение графовым контекстом поверх retrieval-результатов Qdrant.
- Задокументирован воспроизводимый сценарий запуска, импорта и smoke-проверки.
